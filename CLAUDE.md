## 1. 서비스의 본질

### 왜 만드는가

GPS 아트런은 겉보기엔 "달리기"지만, 실제로 해보려는 순간 본체는
**"어떤 경로로 어떻게 달려야 하트/고구마/강아지가 그려지는지 설계하는 일"**이 된다.
지금은 이 설계를 구글 지도와 손품으로 한다. 그려도 공유가 번거롭다.

**지도 위에 점을 찍거나 손가락으로 쭉 긋기만 하면 인도·보행로를 따라 경로가 이어지고, 완성된 그림을 GPX로 내보내 바로 달릴 수 있다.**

### 유저 워크플로우

1. 지도 열리면 **인기 코스·인기 아트런이 맵 위에 옅게 얹혀 있어** 바로 탐색 가능
2. 지도를 탭/클릭하면 **그리기 모드** 진입 — 점 모드와 드래그 모드 모두 지원
3. **점 모드**: 탭할 때마다 점이 추가되며 **인도·보행로를 따라** 자동으로 이어짐
4. **드래그 모드**: 손가락/커서로 쭉 긋는 대로 경로가 샘플링되어 보행로로 스냅
5. 완성 → GPX 다운로드
6. GPX를 카카오맵 등에 넣어 길찾기 경로로 활용 → Strava로 달리며 아트 완성

### 존재 이유 한 줄

> "직선이 아니라 **보행 경로를 따라** 이어진다."

이게 빠지면 서비스는 존재할 이유가 없다. 모든 트레이드오프는 이 원칙에 종속된다.

### 화면 구조 원칙 — "하나의 맵이 곧 앱"

- **화면 뎁스는 하나.** 페이지 라우팅·URL 전환 없음.
- 사이드바·바텀시트·툴바·다이얼로그는 모두 **같은 화면의 오버레이**로만 존재.
- 모달은 입력(코스 이름 등) 최소만.
- 지도가 곧 캔버스이자 갤러리이자 작업 공간.

### 두 갈래의 "추천"

사이드바/바텀시트에 **탭 2개**:

| 탭 | 성격 | 예시 |
|---|---|---|
| **인기 코스** | 뛰기 좋고 풍경 이쁜 일반 러닝 코스 (아트 아님) | 한강 야경 코스, 남한산성 순환, 올림픽공원 한 바퀴 |
| **인기 아트런** | 러닝으로 완성하는 그림 코스 | 고구마런, 하트런, 강아지런, 고래런 |

- 두 리스트 모두 **관리자(프로젝트 오너)가 수동 큐레이션**. 유저 업로드 없음 (MVP 범위 밖).
- 지도 위에는 두 종류 모두 옅게 상시 표시. 색·대시 패턴으로 구분 (인기 코스 = 실선, 인기 아트런 = 대시).

### 핵심 가치 (우선순위 순)

1. **정확한 보행 경로** — 인도·보행로 스냅
2. **두 가지 그리기 모드** — 점(정밀) + 드래그(빠름)
3. **항상 보이는 오버레이** — 지도 자체가 탐색 도구
4. **가볍고 빠름** — 모바일 한 손 조작
5. **영구 무료 운영** — 정적 호스팅

## 2. 선택의 기준 (의사결정 원칙)

모든 기술 선택은 아래 순서로 판단한다.

1. **가벼움** — 초기 번들 < 300KB(gzipped), 모바일 LCP < 1.5s
2. **GitHub Pages 호환** — 정적 파일. API 키는 빌드 타임 주입 + 도메인 allowlist
3. **비용 0원 운영** — 무료 티어 안에서 작동 (디바운스·캐싱·프리컴퓨트)
4. **한국 유저 기준** — 지도 스타일·언어·시드 코스·초기 뷰포트는 한국 기준
5. **보수적 스택** — Vanilla TS + Vite 우선
6. **오버레이 성능** — 수십 개 코스를 동시에 맵에 얹어도 부드러워야 한다.
   이것이 **지도 API 선택의 결정적 기준**이다. Leaflet은 이 규모에서 무겁다 → **MapLibre GL JS + 벡터 타일** 사실상 확정. 타일 서비스(Protomaps/MapTiler/Stadia)만 3종 비교.

## 3. 비즈니스 / 프로덕트 문제

### 3.1 라우팅 쿼터가 병목이다

무료 라우팅 API(OpenRouteService) 한도: **2000 req/day, 40 req/min**.

**점 모드 대책**
- 디바운스 300ms + AbortController로 이전 요청 취소
- 좌표쌍 키 세그먼트 캐시 (in-memory + LocalStorage LRU 500)
- 갤러리 인기 코스는 빌드 타임 프리컴퓨트 → 런타임 호출 0

**드래그 모드 대책** (쿼터 폭발 방지에 특히 중요)
- **거리 기반 샘플링**: 드래그 중 이전 캡처점에서 **30m 이상 이동해야** 새 waypoint 생성
- **단일 드래그 상한**: 40 waypoints (초과 시 추가 캡처 무시 + UI 경고)
- **지연 라우팅**: 드래그 중에는 원본 포인터 궤적을 **직선 점선**으로만 임시 표시. **pointerup 직후** 전체 세그먼트를 한 번에 배치 라우팅
- 점 모드와 세그먼트 캐시 공유

### 3.2 오버레이 성능

수십 개 코스를 동시에 렌더하되 부드러워야 한다.

- **단일 GeoJSON 소스 원칙** — 모든 코스를 한 FeatureCollection으로 묶어 **하나의 map source**에 넣는다. 소스 수 늘리지 않음.
- **줌 기반 가시성**
  - zoom < 10: anchor symbol만 (작은 원 + 이름)
  - zoom ≥ 10: 전체 LineString 렌더
- **프리컴퓨트만** — 오버레이 LineString은 전부 빌드 타임에 결정. 런타임 라우팅 0.
- **단순화된 geometry** — Douglas-Peucker(tolerance ~5m)로 포인트 절반 이하로 감소된 `routeSimplified` 필드를 오버레이용으로 사용. 편집·GPX 저장에는 원본 `route` 사용.

### 3.3 API 키 노출

정적 배포라 키가 번들에 들어간다. **반드시 제공자 대시보드에서 `https://<user>.github.io` 도메인 allowlist 설정**. 키 노출돼도 타 도메인 호출 불가.

### 3.4 모바일 우선

러너는 폰에서 쓴다. 모바일 레이아웃이 **제품의 정식 모드**, 데스크톱은 편의.

## 4. 자주 실수하는 것들 (규칙으로 고정)

> 같은 실수를 두 번 반복하지 않는 것이 목적. 반복 시 여기 추가.

- **좌표 순서 혼동** — GeoJSON `[lng, lat]`, GPX `lat="..." lon="..."`.
  타입 분리: `type LngLat = [number, number]; type GpxPt = { lat: number; lon: number }`. 변환은 `util/coord.ts`에서만.

- **라우팅 프로필 실수** — 반드시 `foot-walking`. enum으로 고정.

- **GPX 포맷** — `<trk>/<trkseg>/<trkpt>`, UTF-8, `xmlns="http://www.topografix.com/GPX/1/1"`. Strava·카카오맵은 `<rte>`보다 `<trk>`를 안정적으로 처리.

- **MapLibre 회색 화면** — 숨겨진 컨테이너(`display:none`)에서 mount 시 0×0. 보이게 된 시점에 `map.resize()` 필수.

- **드래그 모드 중 매 샘플 라우팅 호출** — 쿼터 즉시 소진. 반드시 pointerup까지 대기 후 배치.

- **여러 코스를 각각 별도 source로 생성** — 오버레이 렌더 성능 저하. 단일 FeatureCollection, 단일 source.

- **레이어 순서 뒤집힘** — 오버레이가 그리기 라인 위에 깔리면 인터랙션·가독성 망가짐.
  순서: `basemap < overlay-symbols < overlay-lines < route-pending < route-done < points`

- **편집 모드에서 선택한 코스가 오버레이에도 보임** — 시각적 혼란.
  → "편집하기" 진입 시 해당 feature id를 오버레이 레이어의 `filter`로 제외, 편집 라인으로만 표시.

- **디바운스 취소 누락** — 진행 중 요청은 `AbortController.abort()`.

- **전체 경로 재계산** — 중간 점 하나 수정 시 인접 2 세그먼트만 재계산.

- **Undo/Redo 설계** — 점 리스트 snapshot이 아니라 **연산 히스토리(op log)** 방식.

- **vite base path 누락** — `base: '/art-run/'` 설정 필수, 누락 시 GitHub Pages 404.

- **iOS 터치 vs 클릭** — 지도 내 탭은 MapLibre `click`, 커스텀 마커는 `pointerup`.

- **라우팅 실패 직선 잔존** — 임시 점선이 최종으로 남으면 §1 원칙 위반. 실패 시 에러 + 재시도 버튼.

- **갤러리 런타임 라우팅** — 인기 코스는 프리컴퓨트된 `route`, `routeSimplified` 필수.

- **`#BDEFFC` 단독 라인** — 연한 색이라 지도 타일 위에서 대비 약함. **흰 halo 2px** 기본(라인 2겹 스택).

## 5. 웹사이트 기준

### 5.1 언어
- UI: **한국어 (ko-KR)**. `<html lang="ko">`.
- 영어는 `alt`·`aria-label`에 병기 가능.

### 5.2 폰트
```
-apple-system, BlinkMacSystemFont,
'Pretendard Variable', Pretendard,
'Apple SD Gothic Neo', 'Noto Sans KR',
sans-serif
```
Pretendard CDN 서브셋, `font-display: swap`.

### 5.3 컬러 — 확정

- **액센트 (primary)**: `#BDEFFC` (라이트 아이시 블루)
- **베이스 배경**: `#FFFFFF`
- **잉크 (텍스트)**: `#0B1220`
- **뮤트 (보조 텍스트)**: `#6B7280`
- **위험 (삭제 등)**: `#EF4444`
- **다크모드 없음** (v2 이후)

**경로 렌더 규칙**: 라인은 흰 halo 2px를 바깥에 깔고 `#BDEFFC`를 안쪽에 얹는 **2 레이어 스택**. 오버레이는 `#BDEFFC` + `line-opacity: 0.35`.

### 5.4 그리드 — "하나의 맵" 원칙

- **데스크톱 (≥ 1024px)**: 좌측 사이드바 `320px` (탭 2개: 인기 코스 / 인기 아트런) + 우측 지도 영역. 지도 위 floating toolbar.
- **모바일 (< 1024px)**: 지도 full-bleed + 상단 타이틀 바 + 바텀시트 3단계 (탭 동일).
- **페이지 라우팅 없음.** SPA지만 URL은 해시 상태(`#course=yeouido-sweetpotato`) 정도만.
- 모달은 최소(이름 입력 1개 정도)만 허용.

### 5.5 애니메이션
- 모든 전환 `ease-in-out`, 150~200ms. `map.flyTo` 1200ms.
- `prefers-reduced-motion` 존중.

### 5.6 접근성
- 버튼 최소 44×44px, 간격 8px 이상.
- 모든 상호작용에 `aria-label`.
- 키보드: Tab, Delete(선택 점), Cmd/Ctrl+Z, Cmd/Ctrl+Shift+Z, Esc(모드 종료).

## 6. 진행상황 관리 원칙

- **Plan.md**는 설계도. 설계가 바뀌지 않는 한 건드리지 않는다.
- **Process.md**는 Plan.md의 **섹션 번호·제목을 그대로 미러링**한다.
- 상태 범례: 🟢 완료 / 🟡 진행 중 / ⚪ 미착수 / 🔴 블록
- 부분 구현이면 **어디까지 됐는지** 파일·함수 레벨로 기입.
- **코드 변경이 있을 때마다 Process.md도 같은 커밋에서 업데이트** — 1:1 매핑.

CLAUDE.md 업데이트 규칙
- 새로운 실수 패턴 → §4에 즉시 추가
- 의사결정 기준 변경 → §2 수정 + Plan.md 관련 섹션 점검
- 가볍게 유지. 무거워지면 안 읽힌다.
